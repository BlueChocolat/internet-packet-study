🌐 HTTPとHTML ― ウェブの基本的な仕組み

📡 HTTPとは何か

HTTP（HyperText Transfer Protocol）は、ウェブブラウザとウェブサーバーの間で情報をやり取りするための 「約束事（プロトコル）」です。 わかりやすく言うと、「ブラウザがサーバーに何かをお願いするときのルール」のことです。

世界中の人がそれぞれ勝手な方法でサーバーと会話してしまうと、互換性が取れずウェブは機能しません。 そのため、「こういう書式でお願いし、こういう書式で返答する」という共通ルールをあらかじめ決めておく必要があります。これがHTTPです。

🖥️ ブラウザ
(クライアント)
➡️ リクエスト「このページをください」
⬅️ レスポンス「はい、どうぞ」
🗄️ サーバー

HTTPの基本動作は、この「リクエスト（要求）」と「レスポンス（応答）」の2種類しかありません。 ブラウザが「このページが欲しい」とお願いし、サーバーが「はい、こちらです」と返す ― この単純なやり取りの繰り返しだけで、ウェブは成り立っています。

💡 ステートレスという重要な性質
HTTPには「ステートレス」という特徴があります。これは、1回ごとのリクエストが独立しているという意味です。 サーバーは「さっきのお客さん」と「今来たお客さん」を原則として区別しません。 そのため、ログイン状態などを保つには Cookie や セッション といった追加の仕組みが使われます。
🔐 HTTPとHTTPSの違い
HTTP は通信内容が「平文（ひらぶん）」で流れるため、途中で盗み見ることができてしまいます。 HTTPS はHTTPをTLSという暗号化の層で包んだもので、途中で盗聴・改ざんされても内容が分からないようになっています。現在のウェブでは、HTTPSが標準です。

🚀 ブラウザがウェブサイトを読み込む流れ

たとえば、アドレスバーに https://example.com と入力してエンターを押すと、ほんの一瞬のうちに、裏側では次のような処理が順に行われます。

🔍 DNSによる住所解決

example.com のようなドメイン名は、人間には読みやすいですが、コンピュータはそれだけでは相手の場所を特定できません。 ネットワーク上では、コンピュータは IPアドレス（例：93.184.216.34）という数字の住所で互いを識別しています。

そこで最初に、DNS（Domain Name System）というインターネット上の「電話帳サービス」に問い合わせて、ドメイン名をIPアドレスに変換します。

example.com
➡️ DNSサーバーに問合せ ➡️
93.184.216.34

📞 電話帳で「山田さん」の電話番号を調べるのと同じイメージです。 なお、一度調べた結果はしばらく端末内にキャッシュされ、次回以降は高速化されます。

🔌 サーバーへの接続（TCP / TLS）

IPアドレスがわかったら、ブラウザはそのサーバーに向けてTCP接続を張ります。 TCPは「データを正確に届けるための下地（トランスポート層）」の役割を果たします。

URLが https:// で始まる場合は、さらにTLSハンドシェイクという手順を踏み、暗号通信を開始します。 これによって、通信経路の途中（例えば公衆Wi-Fiなど）で内容を盗み見られたり書き換えられたりすることを防ぎます。

🔓 HTTP（平文）     →     🔒 HTTPS = HTTP + TLS（暗号化）
📨 HTTPリクエストの送信

接続が確立したら、ブラウザはサーバーに対して「このページをください」というHTTPリクエストを送ります。リクエストは大きく以下の3つのパートから構成されています。

リクエストの実例

GET /index.html HTTP/1.1 Host: example.com User-Agent: Mozilla/5.0 (Windows NT 10.0; ...) Accept: text/html,application/xhtml+xml Accept-Language: ja,en;q=0.9 Cookie: sessionid=abc123 # （空行のあとに本文が入ることもある）

主なHTTPメソッド

メソッド意味例
GETデータを取得したいページや画像を表示する
POSTデータを送信・新規登録したいフォーム送信、コメント投稿
PUTデータを置き換えたいプロフィールを丸ごと更新
DELETEデータを削除したい投稿を消す
PATCHデータを一部変更したい名前だけ変更する

ヘッダーの役割

ヘッダーは「補足情報」を伝える欄です。ブラウザの種類（User-Agent）、 受け入れられる形式（Accept）、前回ログイン情報（Cookie）などを添えることで、 サーバーは適切なレスポンスを選んで返すことができます。

📬 サーバーからのレスポンス

サーバーはリクエストを受け取ると、該当するファイルを探したり、データベースを調べたりして、 その結果をHTTPレスポンスとして返します。

レスポンスの実例

HTTP/1.1 200 OK Content-Type: text/html; charset=UTF-8 Content-Length: 1270 Date: Fri, 17 Apr 2026 09:00:00 GMT <!DOCTYPE html> <html><head><title>Example</title></head> <body>...</body></html>

ステータスコードの種類

レスポンスの先頭には必ず3桁のステータスコードがついており、処理結果を一目で伝えます。先頭の数字で大まかな意味が分かれています。

1xx情報（処理継続中）
2xx成功
3xxリダイレクト（別の場所へ）
4xxクライアント側のエラー
5xxサーバー側のエラー

よく見かけるステータスコード

コード意味典型例
200 OK成功通常のページ表示
301 / 302リダイレクトURLが変わった / 一時的な転送
304 Not Modified変更なし（キャッシュ利用）前回と同じなので再利用
400 Bad Requestリクエストが壊れている形式不正
401 Unauthorized認証が必要未ログイン状態
403 Forbidden閲覧権限なし管理者専用ページ
404 Not Foundページが見つからないURLの打ち間違え、削除済み

🧩 HTMLの解析と追加リソースの取得

ブラウザが最初に受け取るのは、通常HTMLファイルです。 しかしHTMLの中には、「この画像も読み込んで」「このCSSも必要」「このJavaScriptも実行して」といった指示が書かれています。

ブラウザはHTMLを上から順に解析しながら、必要な追加ファイルに対してさらにHTTPリクエストを送っていきます。 最近のブラウザは並列に複数のリクエストを飛ばすため、効率よくページを構築できます。

🖥️ ブラウザ
→
GET /index.html （最初の1回）
🖥️ ブラウザ
→
GET /style.css （デザインを整える）
🖥️ ブラウザ
→
GET /script.js （動きをつける）
🖥️ ブラウザ
→
GET /logo.png （画像などの素材）
🖥️ ブラウザ
→
GET /api/user.json （APIからのデータ）
⚡ キャッシュで高速化
一度取得したファイルはブラウザのキャッシュに保存され、次回訪問時はネットワークを使わずに表示されます。 サーバー側も Cache-Control や ETag などのヘッダーで「どれくらいキャッシュして良いか」を指示します。
🎨 ページのレンダリング（描画）

必要な素材が揃うと、ブラウザは以下の工程を経て画面にページを描き出します。

  • DOMツリーの構築：HTMLを解析して「どんな要素がどんな階層で並ぶか」の設計図（DOM）を作る
  • CSSOMツリーの構築：CSSを解析して「何をどんな見た目にするか」を決める
  • レイアウト計算：各要素の位置やサイズを算出する
  • ペイント：実際にピクセルとして画面に描画する
  • JavaScriptの実行：動的な変更を加える（ボタンの挙動、アニメーション、APIとの通信など）

あなたが見ている1枚のページは、実は裏側で DNS問合せ → TCP/TLS接続 → 何十回ものHTTPリクエスト・レスポンス → 解析 → 描画 という長い工程を、ほんの数秒以内にこなした結果なのです。

📄 HTMLとの関係

HTML（HyperText Markup Language）は、ウェブページの中身（テキスト・画像・リンクなど）と構造（見出し、段落、リストなど）を記述するための言語です。 「Markup（マークアップ）」とは、文章に目印（タグ）をつけて意味を与える、という意味です。

🏷️ HTMLのタグの例

<!DOCTYPE html> <html> <head> <title>サンプルページ</title> </head> <body> <h1>大きな見出し</h1> <p>段落のテキストです。</p> <a href="https://example.com">リンク</a> <img src="photo.jpg"> </body> </html>

<h1> は見出し、<p> は段落、<a> はリンク、<img> は画像、 というように、各タグが「ここは何を表す部分か」を意味づけます。

HTTPは「荷物を届ける運び屋」、HTMLは「運ばれる中身」。 HTTPはHTMLだけでなく、画像（JPEG/PNG）、動画（MP4）、CSSファイル、JavaScriptファイル、 APIから返ってくるJSONデータなど、ありとあらゆるデータを運ぶことができます。

📝 覚えておきたいポイント
  • HTMLだけでは「見た目」や「動き」は作れない（CSSやJSと役割分担している）
  • HTTPはHTML専用ではなく、あらゆる形式のデータを運ぶ汎用プロトコル
  • 「ウェブページを見る」＝「裏で大量のHTTPリクエスト・レスポンスが飛び交う」ということ

✨ まとめ

HTTP = ブラウザとサーバーの会話ルール（リクエスト ⇄ レスポンス）
HTML = ウェブページの中身と構造を表す言語
DNS / TLS / CSS / JavaScript がそれぞれ役割を分担

これらが協力しあって、世界中のウェブの世界ができています 🌐
